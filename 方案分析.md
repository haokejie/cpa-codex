# 桌面客户端方案分析（Tauri + Rust + Vue）

## 需求摘要
- 桌面客户端，Rust 后端 + Vue 前端
- 后台长期运行，窗口用于配置
- 允许关闭窗口但进程常驻（托盘）
- 需要登录（账号 + 密码）与登录信息存储
- 后台高频任务：请求接口、查看服务器状态、切换账号
- 需要读写文件，保存几千行 JSON 数据
- 需要开机自启，默认关闭，用户可手动开关
- 不需要“用户注销/未登录仍运行”

## 技术选型结论
- **推荐：Tauri + Vue**
  - Rust 后端天然集成
  - 包体小、性能好、安全模型清晰
  - 适合“后台常驻 + 配置页”模式

## 后台常驻运行方案
- **关闭窗口不退出**：拦截关闭事件，改为 `hide()`
- **托盘管理**：托盘菜单提供“打开配置/退出”
- **后台任务常驻**：Rust 端启动定时任务（如 `tokio::spawn + interval`）
- **单实例**：防止重复启动

## 开机自启
- 使用 `tauri-plugin-autostart`
- 默认关闭，用户在设置页手动开关
- 开关操作需落盘（配置文件）

## 存储策略（核心结论）
- **业务状态数据：SQLite**（推荐）
  - 高并发读写、易查询、支持事务与索引
  - 适配“高频接口请求 + 账号切换 + 状态管理”
- **轻量配置：JSON/YAML 文件**
  - 例如：开机自启开关、最后窗口状态、当前账号

## 登录与敏感信息存储
### 原则
- **不存明文密码到 SQLite/JSON**
- 敏感信息使用系统钥匙串

### 推荐做法
- **系统钥匙串存储**
  - Windows：Credential Manager
  - macOS：Keychain
  - Linux：Secret Service
  - 建议存储：`password` 或 `refresh_token`

- **SQLite 仅存账号元信息**
  - 账号列表、昵称、服务器地址、最后登录时间等

## SQLite 表结构建议（简版）
- `accounts`
  - `id`
  - `username`
  - `server`
  - `last_login_at`

- `session`（如需）
  - `account_id`
  - `access_token`（不建议落盘）

## 风险与注意事项
- IPC 协议不清晰 → 维护成本高
- 数据结构未版本化 → 迁移困难
- 高频请求缺少重试/退避 → 资源占用异常
